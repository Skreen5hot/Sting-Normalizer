# Deterministic Key Normalization Pipeline (v2.1 Hardened)

## 1. Status

This document defines version 2.1 of the Deterministic Key Normalization Pipeline. It supersedes v2 where the two specifications conflict.

The key goals of v2.1 are:

- deterministic execution across conforming ECMAScript 2023 runtimes
- offline-safe JSON-LD portability
- explicit handling of invalid rules and missing optional enrichments
- unambiguous tokenization and warning contracts

## 2. Normative Language

The key words `MUST`, `MUST NOT`, `REQUIRED`, `SHOULD`, `SHOULD NOT`, and `MAY` in this document are to be interpreted as normative requirements.

## 3. Objective

The pipeline transforms a raw machine identifier into a deterministic list of normalized semantic tokens and emits the result as a JSON-LD graph.

The pipeline:

1. normalizes all execution-relevant strings to Unicode NFC
2. applies an explicitly ordered set of canonical rules
3. tokenizes the resulting string with a fixed Unicode-aware rule
4. optionally applies declared token enrichments from task-local inputs
5. emits a portable JSON-LD result with metadata and warnings

The pipeline MUST execute without network access and MUST NOT depend on ambient process state.

## 4. Execution Model

### 4.1 Runtime Baseline

A conforming processor MUST execute according to the ECMAScript 2023 specification.

- Regular expressions MUST use the ECMAScript 2023 RegExp dialect.
- Regex matching MUST operate over UTF-16 code unit sequences.
- The `u` flag MUST be explicitly present when a rule requires Unicode code point semantics or ECMAScript Unicode case folding.
- Replacement strings MUST use standard ECMAScript replacement syntax such as `$1`, `$2`, and `$&`.
- Function-valued replacements are not permitted.

### 4.2 String Normalization Scope

Before validation or execution, the processor MUST normalize every string value that can affect computation to Unicode Normalization Form C (NFC).

This includes:

- `rawIdentifier`
- rule `pattern`
- rule `replacement`
- rule `description`
- dictionary input keys and values
- warning messages generated by the processor

Non-string values MUST remain unchanged.

### 4.3 Canonical Phases

The canonical rule phases are:

- `Preparation`
- `Transformation`
- `Pruning`
- `Enrichment`

`Preparation`, `Transformation`, and `Pruning` operate on the working string.

`Enrichment` operates on the token array after canonical tokenization has completed.

### 4.4 Rule Ordering

Execution order is determined exclusively by the integer `order` property.

- Rules MUST be sorted in strictly ascending numeric `order`.
- Duplicate `order` values are invalid.
- `phase` is mandatory classification metadata and MUST NOT be used as an execution tie-breaker.
- Every `Enrichment` rule MUST have an `order` greater than every non-`Enrichment` rule.

If the pipeline is invalid, the processor MUST stop before applying any rule and MUST emit a failed result as defined in Section 8.3.

### 4.5 Rule Kinds

The canonical rule kinds are:

- `RegexReplace`
- `DictionaryExpand`

If `ruleKind` is omitted, processors MUST treat the rule as `RegexReplace`.

#### 4.5.1 `RegexReplace`

A `RegexReplace` rule MUST define:

- `pattern`
- `flags`
- `replacement`

The processor MUST compile `new RegExp(pattern, flags)` under ECMAScript 2023 semantics. If compilation fails, the pipeline is invalid.

#### 4.5.2 `DictionaryExpand`

A `DictionaryExpand` rule MUST define:

- `inputKey`
- `missingInputBehavior`

`inputKey` names a dictionary object in `NormalizationTask.inputs`.

The dictionary MUST be a JSON object whose keys and values are strings after normalization.

For each token in the current token array:

1. the processor performs an exact key lookup using the token value as-is
2. if a match exists, the mapped string replaces that token
3. the mapped string is then tokenized using the canonical tokenization rule in Section 4.7
4. the resulting replacement tokens are spliced in place

If no dictionary entry exists for a token, the token remains unchanged.

### 4.6 Validation

A task is invalid if any of the following are true:

- `rawIdentifier` is not a string
- `pipeline` is missing
- any rule lacks a required property for its `ruleKind`
- any `order` is not an integer
- any two rules share the same `order`
- any `phase` is outside the canonical phase set
- any regex rule cannot be compiled under ECMAScript 2023
- any `DictionaryExpand` rule refers to an `inputKey` whose provided value is not an object of string-to-string entries
- any `missingInputBehavior` is outside the canonical set: `skipWithWarning`, `fail`

Processors MUST NOT partially execute invalid tasks.

### 4.7 Canonical Tokenization

Canonical tokenization occurs exactly once after the final string-phase rule and before the first enrichment rule.

The processor MUST:

1. split the current working string on one or more Unicode whitespace or punctuation characters using `/[\p{Z}\p{P}]+/u`
2. discard empty results
3. lowercase each token using locale-independent ECMAScript default case conversion

Additional heuristic stripping is not permitted.

Implications:

- hyphens, underscores, apostrophes, and other punctuation are treated as delimiters because they are consumed by the split expression
- symbols in Unicode category `S` are not delimiters unless removed by an earlier rule
- digits remain part of tokens unless removed by an earlier rule

### 4.8 Missing Optional Inputs and Degraded Execution

The processor MUST use only data provided in the task payload.

If a rule declares an `inputKey` that is not present in `NormalizationTask.inputs`, the processor MUST follow `missingInputBehavior`:

- `skipWithWarning`: skip that rule and append a warning
- `fail`: stop execution and emit a failed result

For `skipWithWarning`, the warning MUST include:

- `warningCode`: `MISSING_OPTIONAL_INPUT`
- `message`
- `severity`: `warning`
- `ruleId`
- `phase`
- `missingInput`

Skipping a rule MUST NOT skip other rules unless those rules independently require missing inputs.

### 4.9 No Network Access

A conforming processor MUST make zero network calls during normalization, including JSON-LD context resolution.

If an implementation supports external context IRIs, they MUST resolve only against a bundled local registry or preloaded document loader. Implementations MUST NOT dereference context IRIs over the network.

The canonical examples in this document use inline contexts to avoid ambiguity.

## 6. Canonical Data Contracts

### 6.1 `NormalizationTask`

A task MUST contain:

- `@type: "NormalizationTask"`
- `rawIdentifier`
- `pipeline`

A task MAY contain:

- `inputs`

### 6.2 `PipelineDefinition`

A pipeline definition MUST contain:

- `@type: "PipelineDefinition"`
- `rules`

A pipeline definition SHOULD define `@id` when reuse or traceability is required.

### 6.3 `Rule`

A rule MUST contain:

- `@type: "Rule"`
- `phase`
- `order`

A rule MUST satisfy the property requirements of its `ruleKind`.

### 6.4 `Warning`

A warning object MUST contain:

- `@type: "Warning"`
- `warningCode`
- `message`
- `severity`

A warning SHOULD contain `ruleId`, `phase`, and `missingInput` when applicable.

### 6.5 `SemanticTokenList`

The result object MUST contain:

- `@type: "SemanticTokenList"`
- `sourceIdentifier`
- `normalizedString`
- `tokens`
- `metadata`

`metadata` MUST contain:

- `status`
- `executionEnvironment`
- `warnings`

`metadata` SHOULD contain `appliedPipeline` when the pipeline has an `@id`.

## 9. Conformance Requirements

An implementation conforms to v2.1 only if it satisfies all of the following:

- it normalizes all execution-relevant strings to NFC before validation and execution
- it validates tasks before applying rules
- it never uses JSON array order as an implicit execution contract
- it executes rules solely by ascending unique `order`
- it performs canonical tokenization exactly as specified in Section 4.7
- it performs zero network calls during normalization
- it emits warnings in the canonical object shape
- it emits deterministic success or failure results for the same task payload in any conforming ECMAScript 2023 runtime

## 10. Compatibility Notes

Compared with v2, version 2.1 makes the following hardening changes:

- the canonical JSON-LD form is now inline and fully term-defined
- `appliedPipeline` and `executionEnvironment` are explicitly defined
- duplicate rule orders are invalid
- invalid regexes and invalid task shapes have defined failure behavior
- enrichment dependencies are declared in the rule schema instead of implied by prose
- degraded execution behavior is normative and machine-readable
- tokenization no longer relies on extra heuristic punctuation stripping
